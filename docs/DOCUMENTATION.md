# Documentation System

Price Games keeps its documentation honest with a small amount of machinery so
that, as the code changes, the docs are visibly tracked rather than silently
rotting. This page describes the conventions; `scripts/check-docs.mjs` enforces
them and CI runs it on every push.

## The three guarantees

1. **Every doc declares its own metadata** (frontmatter) — including when it was
   last reviewed and which code it describes.
2. **The index can't drift.** `docs/README.md` is generated from that metadata;
   adding or renaming a doc without updating the index fails the check.
3. **Staleness is surfaced.** If code a doc points at changed *after* the doc's
   `last_reviewed` date, the check warns — a nudge to re-read and bump the date.

## Frontmatter schema

Every Markdown file under `docs/` (except the generated `docs/README.md`) starts
with a YAML frontmatter block:

```yaml
---
title: API Reference
status: stable            # stable | beta | draft | deprecated
last_reviewed: 2026-06-03 # YYYY-MM-DD — bump when you verify the doc vs the code
owner: core               # owning area, NOT a person: core | infra | growth | streamer | web
audience: contributor     # contributor | operator | user | all
category: api             # drives the index grouping (see below)
summary: All REST endpoints — public, authenticated, and admin.
related_code:             # paths/dirs this doc describes; used for staleness checks
  - apps/server/src/routes/
  - packages/shared/src/
---
```

All keys except `related_code` are required (an empty `related_code` list is
allowed but discouraged — without it, staleness can't be detected).

### Fields

| Field | Meaning |
|---|---|
| `title` | Human title shown in the index. |
| `status` | `stable` (trustworthy), `beta` (mostly accurate, evolving), `draft` (WIP), `deprecated` (kept for history, not maintained). |
| `last_reviewed` | The date a human last confirmed the doc matches the code. Bump it in the same PR that touches the feature. |
| `owner` | The **area** responsible, never a personal name or email (keeps the public repo free of PII). |
| `audience` | Who the doc is for. `operator` docs assume production access; `contributor` docs assume only a local checkout. |
| `category` | One of the categories below; controls which section of the index the doc lands in. |
| `summary` | One sentence; reused verbatim as the index row. |
| `related_code` | Globs/paths the doc explains. The linter flags the doc as stale when any of these changed after `last_reviewed`. |

### Categories

`getting-started`, `architecture`, `game-design`, `api`, `operations`,
`features`, `streamer`, `extension`, `analytics`, `testing`, `meta`.

To add a category, extend the `CATEGORIES` list in `scripts/check-docs.mjs`.

## The generated index

`docs/README.md` has a hand-written header followed by an autogenerated block
between `<!-- BEGIN AUTOGEN-INDEX -->` and `<!-- END AUTOGEN-INDEX -->`. The
block is a per-category table of every doc with its status, review date, and
summary. **Don't edit between the markers** — edit the doc's frontmatter and
regenerate:

```bash
npm run docs:index   # rewrite the autogen block from frontmatter
```

## Running the checker

```bash
npm run docs:check    # validate frontmatter, index sync, and links (CI runs this)
npm run docs:index    # regenerate the index block in place
node scripts/check-docs.mjs --strict   # also fail on staleness warnings
```

What it checks:

- **Frontmatter** present and valid (required keys, enum values, date format).
- **Index sync** — the committed autogen block matches what the frontmatter
  would produce. A mismatch is an error; run `npm run docs:index`.
- **Relative links** — every relative Markdown link in docs and the root
  governance files (`README`, `CONTRIBUTING`, `SECURITY`, …) resolves to a real
  file. Links that escape the repo (GitHub repo-relative, e.g. to the issues
  page) are left to GitHub.
- **Staleness** (warning) — for each `related_code` path, the latest git commit
  date is compared to `last_reviewed`.

It has **no external dependencies** (the frontmatter parser is hand-rolled), so
it runs in CI without an extra install step.

## Adding or changing a doc

1. Create `docs/MY_TOPIC.md` with the frontmatter block above.
2. Pick the right `category`; set `last_reviewed` to today and list the
   `related_code` paths.
3. Run `npm run docs:index` to add it to `docs/README.md`.
4. Run `npm run docs:check` and fix anything it reports.
5. When you change a feature, update its doc **and** bump `last_reviewed` in the
   same PR — see the Documentation checklist in `CONTRIBUTING.md`.
